## 2. RAG 담당이 제안하는 구조 (한 repo, 데이터 3분할)

```
ai_moneyingbot_rag_agent/            # 단일 repo 유지 (코드 의존성 때문)
  src/
    rag/        # RAG 소유: 검색·리랭크·eval
    trading/    # 트레이딩봇 소유: 백테스트·모의투자
    archive/    # 아카이브봇 소유: 수집·색인·코퍼스 빌드
  data/
    corpus/     # 아카이브봇이 생산 -> RAG·트레이딩봇 읽기전용   [gitignore, 수동이관]
    trading/    # 트레이딩봇만 쓰기 (paper_trading.db, ohlcv.db) [gitignore]
    archive/    # archive.db(16GB) 등 아카이브봇 전용, PC only    [gitignore]
```

**절대 규칙 (이거 하나가 충돌 공포를 없앤다):**
> **어떤 봇도 다른 봇 소유의 DB 파일에 쓰지 않는다.** `corpus`는 모두 읽기 전용, 각자의 쓰기는 자기 `data/<봇>/`에만.

## 3. RAG 담당의 약속 (지킬 계약)

- **안정된 검색 인터페이스 제공** — 트레이딩봇은 아래 시그니처만 의존하면 된다. 예고 없이 안 깬다:
  ```python
  # src/rag/ 에서 노출
  retrieve_then_rerank(query: str, top_k: int, fetch_k: int = 50) -> list[Chunk]
  # embed_query(voyage-3-large) -> qdrant -> rerank-2, 이미 검증됨(recall@1 0.80)
  ```
- RAG는 `data/corpus/`를 **읽기만** 한다. `data/trading`·`data/archive`는 안 건드린다.
- 검색 품질 변경(리랭커 교체, fetch_k 튜닝)은 인터페이스 시그니처를 유지한 채 내부에서만.

## 4. 아카이브봇 담당에게 — 확정 요청

아카이브봇은 코퍼스(qdrant + mentor.db)의 **유일 생산자**다. RAG·트레이딩봇이 산출물을 읽으므로 사실상 "데이터 API"를 정의한다.

1. **인터페이스 동결/버전 요청:** 아래를 함부로 바꾸지 말고, 바꾸면 알릴 것.
   - qdrant: 컬렉션명(`goodmorning_chunks`), 차원(1024), payload 필드(`text`, `article_id` 등)
   - mentor.db: 스키마(`articles`, `articles_fts`, 컬럼명)
   - -> 하나라도 바뀌면 RAG 검색과 트레이딩봇 룰 마이닝이 동시에 깨진다.
2. **재생성 알림 규약:** qdrant/mentor.db를 재수집·재색인하면 RAG·트레이딩봇 사본이 낡는다. **버전 스탬프**(예: `data/corpus/VERSION` 파일에 생성일+커밋)를 남겨 소비자가 "내 사본이 최신인지" 판단하게 할 것.
3. **확인:** `archive.db`(16GB)는 계속 PC 전용·수동이관 제외 — 맞나?

## 5. 트레이딩봇 담당에게 — 확정 요청

트레이딩봇은 RAG 파이프라인의 **소비자**이자 모의투자 상태의 **유일 생산자**다.

1. **RAG에서 뭘 필요로 하나?** §3의 `retrieve_then_rerank`면 충분한가, 아니면 (원문 전체 열람용) `mentor.db` 직접 읽기도 쓰나? 의존하는 표면을 명시할 것 — 그래야 RAG가 안 깬다.
2. **쓰기 격리 확인:** 모의투자·백테스트·OHLCV는 전부 `data/trading/`에만 쓰고, `data/corpus`엔 **절대 안 쓴다** — 동의하나?
3. **모의투자는 돌아가는 프로세스다 -> 어느 기계에서?** PC가 데이터·운영 담당이니 **모의투자도 PC 상시 실행**을 제안한다. 노트북이 낡은 코퍼스 사본 위에서 병행 실행하면 divergence 난다. 동의하나?
4. **동시성:** 모의투자 상태 DB는 `PRAGMA journal_mode=WAL`로 — 루프가 쓰는 동안 백테스트/조회가 읽어도 락 안 걸리게.
